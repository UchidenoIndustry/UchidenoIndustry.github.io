BP STEP 07

タグの変化を待つ

UE 5.5 版

WaitGameplayTagAdded / WaitGameplayTagRemoved で ASC のタグ変化に反応するアビリティを実装する。

このステップのゴール

これまで作ったアビリティは「発動 → 処理 → 終了」という一方向の流れでした。 このステップでは Wait* ノード（Ability Tasks） を使い、 アビリティが途中で「タグが付くまで待つ」「タグが外れるまで待つ」処理を書けるようにします。

新しく GA_StunWatcher というアビリティを作ります。 このアビリティはゲーム開始時から常時動作し、Status.Stunned タグが 追加されたとき・削除されたときを自動で検知します。

  • GA_StunWatcher Blueprint を作成する
  • Wait Gameplay Tag Added でスタン発生を検知し「スタン検知！」を表示する
  • Wait Gameplay Tag Removed でスタン解除を検知し「スタン解除！」を表示する
  • E キーを押すたびに 2 つのメッセージが順番に表示されることを確認する

GA_StunWatcher は「スタンがどうやって付与されたか」を知りません。 BP STEP 04 の Add Loose Gameplay Tags でも、 BP STEP 06 の GE_Stun でも、タグが変化すれば同じように反応します。 これが GameplayTag によるゆるやかな連携（疎結合）の便利さです。

GA_StunWatcher を作成する

1
GA_StunWatcher Blueprint を作成する

Content Browser（コンテンツブラウザ）で右クリック → 「Blueprint Class（ブループリントクラス）」を選択。 親クラスの検索欄に 「GameplayAbility」 と入力して選択し、 名前を GA_StunWatcher にして作成します。

GA_StunWatcher を開くと、イベントグラフに Event ActivateAbility がすでに配置されています。

アビリティのグラフを組む

通常のアビリティは「発動 → Commit → End」と一直線に終わります。 GA_StunWatcher は「発動 → Commit → ループで待ち続ける」という構造にします。 EndAbility は呼ばず、アビリティを常時起動したままにします。

2
Commit とカスタムイベントを接続する

イベントグラフの空白を右クリックして 「Commit Ability」 ノードを配置し、 Event ActivateAbility の実行ピンに繋ぎます。

次に空白を右クリックして 「Add Custom Event...」を選択し、 イベント名を WatchForStun にします。 Commit Ability の実行出力ピンを WatchForStun に繋ぎます。

Event ActivateAbility → Commit Ability → WatchForStun（カスタムイベント）
EndAbility を呼ばない理由

GA_StunWatcher はゲーム中ずっとタグの変化を監視します。 EndAbility を呼ぶとアビリティが終了して監視が止まります。 常時動作させるため、ここでは EndAbility を使いません。

3
Wait Gameplay Tag Added でスタンを検知する

WatchForStun カスタムイベントの実行ピンから続きを作ります。 グラフの空白を右クリックして 「Wait Gameplay Tag Added」 で検索し、ノードを配置します。 WatchForStun の実行ピンを繋ぎます。

Wait Gameplay Tag Added の Tag ピンに Status.Stunned を設定します。

Wait Gameplay Tag Added の 「Added」出力ピンに Print String（文字列: "スタン検知！"）を繋ぎます。

WatchForStun → Wait Gameplay Tag Added
Tag: Status.Stunned [Added] → Print String
"スタン検知！"
タグが付くまでここで待機。付いた瞬間に「Added」ピンから続く。
Wait Gameplay Tag Added の動作

このノードに実行が届くと、Status.Stunned タグが ASC に追加されるまでアビリティが待機します。 タグが追加された瞬間に 「Added」 ピンから実行が再開されます。

4
Wait Gameplay Tag Removed で解除を検知し、ループさせる

Print String「スタン検知！」の実行出力ピンから続きを作ります。 グラフの空白を右クリックして 「Wait Gameplay Tag Removed」 を検索して配置します。 Tag ピンに Status.Stunned を設定します。

Wait Gameplay Tag Removed の 「Removed」出力ピンに Print String（文字列: "スタン解除！"）を繋ぎます。

Print String「スタン解除！」の実行出力ピンを WatchForStun カスタムイベントに繋いでループを作ります。 これで、スタン解除後も次のスタンを待ち続けます。

グラフ全体の流れ
Event ActivateAbility → Commit Ability → WatchForStun
WatchForStun → Wait Tag Added
Status.Stunned [Added] → Print "スタン検知！"
↓
Wait Tag Removed
Status.Stunned [Removed] → Print "スタン解除！" → WatchForStun ↺
EndAbility は接続しない。解除後も次のスタンを待ち続ける。

ブループリントをコンパイルして保存してください。

BeginPlay でアビリティを常時起動する

5
キャラクターの BeginPlay に追加する

BP_ThirdPersonCharacter（使用中のキャラクター Blueprint）のイベントグラフを開きます。 既存の Event BeginPlay に、GA_StunWatcher の登録と起動を後ろに追加します。

Give Ability（Ability Class: GA_StunWatcher）を配置し、 その後に Try Activate Ability By Class（In Ability to Activate: GA_StunWatcher）を繋ぎます。 どちらも右クリック検索で配置でき、Target に AbilitySystem が自動接続されます。

既存のノードは残す

BP STEP 03 以降で追加した GA_MyFirstAbility の Give Ability は 削除せずそのまま残してください。 GA_StunWatcher の Give Ability → Try Activate Ability By Class を その後ろに追加します。

TryActivateAbilityByClass で即座に起動する

GA_StunWatcher はゲーム開始直後から監視を始める必要があります。 BeginPlay で Try Activate Ability By Class を呼ぶことで、 ゲーム開始後すぐにアビリティが起動してタグの変化を待機し始めます。

ブループリントをコンパイルして保存してください。

動作確認

6
スタン検知・解除を確認する

プレイを開始して以下の順で操作します。

  • E キーを押す → 「Stunned! (3s)」（bp06 の Print String）と「スタン検知！」が表示される
  • 3 秒待つ → 「スタン解除！」が表示される
  • E キーをもう一度押す → 再び「Stunned! (3s)」と「スタン検知！」が表示される（ループ動作）
メッセージが表示されない場合

BeginPlay で Give Ability と Try Activate Ability By Class の両方が GA_StunWatcher に対して接続されているか確認してください。 片方だけでは動作しません。

Wait* ノードとは

Wait Gameplay Tag Added / Wait Gameplay Tag Removed は Ability Task という仕組みで実装された非同期ノードです。 アビリティ内で「何かが起きるまで待つ」処理を書くことができます。

ノード 待つ条件 発火する出力ピン
Wait Gameplay Tag Added 指定タグが ASC に追加されるまで Added
Wait Gameplay Tag Removed 指定タグが ASC から削除されるまで Removed
Ability Task とは

Ability Task は GameplayAbility 内で非同期処理を行うための仕組みです。 Wait Gameplay Tag Added 以外にも Wait Input Pressed（入力を待つ）・Wait Delay（時間を待つ）・ Wait Target Data（ターゲット選択を待つ）など多くの種類があります。 グラフで「Wait」と検索すると一覧が表示されます。

Wait* パターンとサンプルプロジェクト

Lyra などの公式サンプルプロジェクトを見ると、アビリティのグラフに Wait～ から始まるノードが多数登場します。 これは「アビリティが能動的に何かを待つ」という GAS 特有のパターンです。

このパターンの特徴は 疎結合 です。 GA_StunWatcher は「スタンを付与する側」を知りません。 Add Loose Gameplay Tags で付与しても、GE_Stun で付与しても、 将来 C++ で付与しても、Status.Stunned タグが変化する限り同じように動作します。 「タグが変化したという事実」だけを介して連携するため、 付与側と検知側を独立して変更できます。

サンプルプロジェクトを読むために

Wait* ノードに慣れると、GAS を使ったプロジェクトのアビリティが読みやすくなります。 「このアビリティはここで何を待っているのか」を読み解くのが GAS 理解の鍵です。 次のステップでは Wait Gameplay Event を使い、 アビリティ間でのイベント連携を学びます。

うまくいかないときの確認ポイント

  • 「スタン検知！」が表示されない → BeginPlay で Give Ability と Try Activate Ability By Class に GA_StunWatcher が設定されているか
  • 「スタン検知！」が表示されない → Wait Gameplay Tag Added の Tag ピンに Status.Stunned が設定されているか
  • 「スタン解除！」が表示されない → Wait Gameplay Tag Removed の Tag ピンに Status.Stunned が設定されているか
  • 2 回目以降のスタンで検知しない → Print「スタン解除！」の後に WatchForStun カスタムイベントへの接続があるか（ループになっているか）
  • すべてのブループリントをコンパイル・保存済みか
チュートリアル通りにならない場合

このチュートリアルは UE 5.5 で動作確認しています。別バージョンをお使いの場合は UE 5.5 で試してみてください。

まとめ

Wait Gameplay Tag Added / Wait Gameplay Tag Removed を使うと、 アビリティが「タグの変化を待つ」ことができます。
タグを付与する手段を問わず反応できるため、システムを疎結合に保てます。 これが GameplayTag の「管理が便利」という実感の核心です。
