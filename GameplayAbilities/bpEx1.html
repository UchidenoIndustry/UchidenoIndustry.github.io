BP EX 01

コンボ先行入力

UE 5.5 版

BP STEP 08 のコンボに「少し早めに押しても繋がる」先行入力を追加し、プロ品質の操作感を実現する。

このページのゴール

BP STEP 08 で作ったコンボは、AN_ComboReady が発火してから 0.5 秒のウィンドウ内に Q を押す「反応型」でした。 このページでは 先行入力（プリインプット） を追加します。 攻撃モーション中に Q を押しておくだけで、タイミングが来たときに自動的に次のコンボへ繋がります。

  • AnimNotify State（範囲通知）を使い、「入力受付ウィンドウ」と「実行タイミング」を分離する
  • Character BP に bComboBuffered フラグを追加し、受付ウィンドウ内の Q 入力を記憶する
  • GA 内でフラグを確認し、立っていれば実行タイミングで自動的に次段を起動する
  • フラグが立っていない場合は bp08 の GE_ComboWindow パス（反応型）にフォールバックする
前提

BP STEP 08 が完成していることが前提です。このページは bp08 の GA・アニメーション・Character BP を改修する形で進めます。

なぜ先行入力が必要か

人間の視覚的反応速度は平均 150〜200ms です。 攻撃アニメーションを「見てから」ウィンドウに反応するのは、短いウィンドウではほぼ不可能です。

bp08（反応型） bpEx1（先行入力）
Q を押すタイミング AN_ComboReady 発火後に押す アニメーション中に早めに押しておく
失敗のしやすさ ウィンドウを見てから反応が必要 アニメーションのリズムで押せばよい
フォールバック なし（ウィンドウを逃すと Combo1 から再スタート） バッファなしでも GE_ComboWindow パスで反応型入力も可能

商業タイトルではほぼすべてのアクションゲームで先行入力が実装されています。 GAS でこれを実現するには AnimNotify State による入力受付区間の定義と、フラグ変数によるバッファリングを組み合わせます。

仕組みを理解する

AnimNotify State とは

bp06 / bp08 で使った AN_AttackHit / AN_ComboReady は アニメーションの特定フレームで一瞬だけ発火する AnimNotify（点）です。 AnimNotify State（範囲）は開始フレームと終了フレームの両方を持ち、 Received Notify Begin と Received Notify End の 2 つのイベントをオーバーライドできます。

モーション開始

入力受付ウィンドウ
Q を押すとバッファに記憶
GE_ComboWindow

ウィンドウ閉鎖

ANS_ComboBuffer の Begin（開始フレーム）から End（終了フレーム）までが入力受付ウィンドウです。 End が「実行タイミング」を兼ね、Event.Combo.Ready を送信します。 これにより、End は bp08 の AN_ComboReady と同じ役割を果たします。

全体フロー（先行入力パス）

バッファ済みの場合（早めに Q を押したとき）
AM_Combo1 再生中 → ANS Begin 発火 → Status.ComboInput.Accept タグ付与
↓ （この間に）Q Pressed
Character BP: Accept タグあり → bComboBuffered = true
ANS End 発火 → Event.Combo.Ready 送信 → GA が受信
→ Accept タグ除去 → bComboBuffered チェック → true → GA_Combo2 発動（自動）
バッファなしの場合（ANS End 後に Q を押したとき）
ANS End 発火 → GA が受信 → bComboBuffered = false → GE_ComboWindow1 付与（フォールバック）
↓ Q Pressed（0.5 秒以内）
cascading TryActivate → GA_Combo2 発動（反応型）
どちらのパスでも GA_Combo2 が発動し、Cancel Abilities with Tag で GA_Combo1 が終了する。

設計する

新しく追加するタグ

タグ 用途 管理者
Event.Combo.BufferStart ANS_ComboBuffer の Begin（開始）が送る Gameplay Event。GA が受信して入力受付ウィンドウを開く。 ANS_ComboBuffer
Status.ComboInput.Accept 入力受付ウィンドウが開いている間だけ ASC に付与される Loose タグ。Character BP がこのタグを見て Q 入力をバッファするかどうか判断する。 GA（Loose タグ）
bComboBuffered 変数について

タグだけでバッファ管理をすることも可能ですが、このページでは Character BP の bool 変数 bComboBuffered を使います。「入力を記憶している」という状態が変数として明確に見えるため、理解しやすい構成です。

既存タグの再利用

ANS_ComboBuffer の End は bp08 の AN_ComboReady と同じ Event.Combo.Ready を送信します。 GA 側の Wait Gameplay Event ノードはそのまま使えます。 「実行タイミングを送るタグ」を変えずに済むため、改修箇所が最小になります。

STEP 1 — ANS_ComboBuffer を作る

1
タグを 2 つ新規作成する

任意の GameplayAbility の Class Defaults → AssetTags の「+」から以下の 2 タグを作成します。

  • Event.Combo.BufferStart
  • Status.ComboInput.Accept

作成後、AssetTags の一覧から削除してかまいません。タグは DefaultGameplayTags.ini に保存されているため、削除しても消えません。

2
ANS_ComboBuffer Blueprint を作成する

Content Browser で右クリック → Blueprint Class → 親クラスに AnimNotifyState を指定し、 名前を ANS_ComboBuffer とします。

作成した ANS_ComboBuffer を開き、My Blueprint パネル → FUNCTIONS → Override（+） から Received Notify Begin と Received Notify End を両方追加します。

Received Notify Begin のグラフ

Received Notify Begin
(Mesh Component, Animation) → Mesh Component → Get Owner → Send Gameplay Event to Actor
Actor: GetOwner / Event Tag: Event.Combo.BufferStart
入力受付ウィンドウの開始を GA に通知する。

Received Notify End のグラフ

Received Notify End
(Mesh Component, Animation) → Mesh Component → Get Owner → Send Gameplay Event to Actor
Actor: GetOwner / Event Tag: Event.Combo.Ready
実行タイミングを GA に通知する。bp08 の AN_ComboReady と同じ Event Tag を使う。
Send Gameplay Event to Actor の Actor ピン

Mesh Component → Get Owner を必ず繋いでください。未接続のままだと ASC が見つからずサイレントに失敗します。

STEP 2 — アニメーションを更新する

3
AM_Combo1 と AM_Combo2 の Notifies を置き換える

AM_Combo1 と AM_Combo2 をそれぞれ開きます（bp08 では AN_ComboReady が Notifies トラックに配置されています）。

  1. Notifies トラックの AN_ComboReady を右クリック → Delete（削除）。
  2. Notifies トラックを右クリック → Add Notify State（通知ステートを追加） → ANS_ComboBuffer を選択。
  3. 配置された ANS_ComboBuffer の End 位置（右端）を、削除した AN_ComboReady があったフレームに合わせます。
  4. Begin 位置（左端）を End より 5〜10 フレーム手前に設定します。これが入力受付ウィンドウの長さになります。
Begin の位置の目安

Begin を早くするほど先行入力できる時間が長くなります。アニメーションのリズムに合わせて調整してください。攻撃モーション全体の前半から受け付けると自然な操作感になります。

GA_Combo3 のアニメーションは変更不要

GA_Combo3 は最後の段なので次のコンボへのバッファは必要ありません。AM_Combo3 の AN_ComboReady は配置していない想定ですが、もし配置していても削除する必要はありません。

STEP 3 — Character BP を更新する

4
bComboBuffered 変数を追加する

キャラクター Blueprint を開き、My Blueprint パネル → VARIABLES の「+」から変数を追加します。

変数名型デフォルト値
bComboBufferedBooleanfalse
5
Q キーのロジックを更新する

bp08 で実装した Q キーの cascading TryActivate ロジックを以下のように変更します。 最初に Status.ComboInput.Accept タグの有無を確認し、 タグがある間（＝入力受付ウィンドウ内）はバッファに記憶します。

Q Pressed → Has Matching Gameplay Tag
Target: AbilitySystem コンポーネント
Tag: Status.ComboInput.Accept → Branch
True → Set bComboBuffered = true
False → Try Activate Ability By Class（GA_Combo3） → Return Value → Branch
False → Try Activate Ability By Class（GA_Combo2） → Return Value → Branch
False → Try Activate Ability By Class（GA_Combo1）
Accept タグなし（ウィンドウ外）= bp08 と同じ cascading TryActivate。Accept タグあり（ウィンドウ内）= バッファに記憶して即座の発動はしない。
Has Matching Gameplay Tag の Target

Character BP では My Blueprint パネルの Components から AbilitySystem をドラッグしてノードを作り、Has Matching Gameplay Tag の Target ピンに繋ぎます。

STEP 4 — GA_Combo1 を改修する

変更点は 3 カ所です。

6
ActivateAbility の先頭でバッファをリセットする

Event ActivateAbility ノードの直後に以下のノードを追加します。 前のコンボ段が終わった際に残っているかもしれないフラグをクリアします。

Event ActivateAbility → Get Owning Actor → Cast To [キャラクタークラス] → Set bComboBuffered = false → Commit Ability → Sequence
キャラクタークラスは自分のプロジェクトのキャラクター Blueprint 名に合わせてください（例: BP_ThirdPersonCharacter）。Cast の結果ピンはこの後も使います。
Cast To の使い方

GA のグラフで右クリック → 「Cast To」 と検索し、自分のキャラクター BP クラス名を選びます。Get Owning Actor の戻り値を Object ピンに繋ぎます。Cast に成功すると As [クラス名] ピンからキャラクター BP の変数にアクセスできます。

7
Sequence に Then 1（BufferStart 処理）を追加する

既存の Sequence ノードを右クリック → Add pin（ピンを追加）して Then 1 を作ります。 元の Then 1（Wait Gameplay Event Event.Combo.Ready）は Then 2 にずれます。

Then 0 → Play Montage and Wait（AM_Combo1） → [各出力] → End Ability
新規Then 1 → Wait Gameplay Event
Event.Combo.BufferStart / Only Trigger Once: true Event Received → Add Loose Gameplay Tags
Status.ComboInput.Accept / Actor: Self
Then 2 → Wait Gameplay Event（Event.Combo.Ready）…
ANS Begin が発火するとバッファウィンドウが開く。Character BP の Q 入力が Status.ComboInput.Accept タグを見てバッファするかどうかを判断できるようになる。
Add Loose Gameplay Tags の Actor ピン

Actor ピンに Self ノードを明示的に繋いでください。未接続だとタグの更新がサイレントに失敗します。

8
Then 2（Event.Combo.Ready 処理）にバッファチェックを追加する

元の Then 1 だった部分（Wait Gameplay Event Event.Combo.Ready）を以下のように拡張します。 Accept タグの除去、バッファチェック、発動または GE 付与の分岐を追加します。

Then 2 → Wait Gameplay Event
Event.Combo.Ready / Only Trigger Once: true
Event Received → Remove Loose Gameplay Tags
Status.ComboInput.Accept / Actor: Self → Cast result → Get bComboBuffered → Branch
True → Cast result → Set bComboBuffered = false → Try Activate Ability By Class（GA_Combo2）
False → Apply Gameplay Effect to Owner
GE_ComboWindow1
「Cast result」はSTEP 6 で作成した Cast To ノードの「As [クラス名]」ピンを再利用します。
Cast result の再利用

STEP 6 で Event ActivateAbility の直後に作った Cast ノードの As [クラス名] ピンから直接ワイヤーを引き出して変数にアクセスできます。新しい Cast ノードは不要です。

GA 内から TryActivateAbilityByClass を呼ぶ

GA Blueprint のグラフで右クリック → 「Try Activate Ability By Class」 を検索すると、キャラクターの AbilitySystem コンポーネントが自動接続された状態で出てきます。In Ability to Activate ピンに GA_Combo2 を指定してください。

STEP 5 — GA_Combo2 を改修する

GA_Combo1 と全く同じ変更を GA_Combo2 に適用します。異なるのはモンタージュ名・GE 名・発動先のみです。

変更箇所 GA_Combo1 の設定値 GA_Combo2 の設定値
STEP 6：バッファリセット 同じ（Set bComboBuffered = false） 同じ
STEP 7：Then 1 で付与するタグ Status.ComboInput.Accept Status.ComboInput.Accept（同じ）
STEP 8：Then 2 の True 側 TryActivate GA_Combo2 GA_Combo3
STEP 8：Then 2 の False 側 GE GE_ComboWindow1 GE_ComboWindow2
GA_Combo3 は変更不要

GA_Combo3 は最終段なので次のコンボはありません。バッファ関連の変更を加える必要はありません。

動作確認

Play（プレイ）して以下のシナリオを確認します。

  • 先行入力パス： Combo1 再生中（ANS 範囲内）に Q を押すと、ANS End のタイミングで自動的に Combo2 に切り替わる
  • 反応型パス： ANS End 後（GE_ComboWindow 中）に Q を押しても Combo2 に繋がる（フォールバック）
  • ウィンドウ外： Combo1 終了後に Q を押すと Combo1 から再スタートする
  • Combo1 → Combo2 → Combo3 の 3 連撃： 各段で先行入力して全段完走できる
  • 早押しすぎ： ANS Begin より前（まだモーション序盤）に Q を押しても Accept タグがなく、cascading TryActivate が走り Combo1 から再スタートする
バッファウィンドウの調整

ANS_ComboBuffer の Begin 位置を早めると先行入力できる時間が長くなり、遅めると短くなります。アニメーションのテンポに合わせて Sequence エディタ（Animation Montage エディタ）で微調整してください。

キャンセル時の状態確認

GA_Combo1 が GE_ComboWindow1 を付与した後に何らかの理由でコンボが中断された場合、Status.ComboInput.Accept タグが残ることはありません（GA が終了すると Remove Loose Tags は呼ばれないため、STEP 6 のバッファリセットタイミングでタグが残っていても次の ActivateAbility で正しくリセットされます）。ただし GE_ComboWindow1 は Duration まで ASC に残ります。必要に応じて GA の Event OnEndAbility で GE を手動除去してください。

まとめ

今回追加した先行入力の仕組みを整理します。

  • AnimNotify State：アニメーションの「範囲」を定義する通知。Begin と End で開始・終了を別々に処理できる。
  • 入力バッファリング：キー入力を即座に実行せず変数（bComboBuffered）に記憶し、適切なタイミングで取り出して使う手法。
  • 先行入力：入力受付ウィンドウを実行タイミングより早めに開くことで、人間の反応速度に依存しない操作感を実現する。商業タイトルでは標準的な実装。
  • フォールバック設計：バッファがない場合は既存の GE_ComboWindow パス（反応型）に落ちるため、先行入力を追加しても bp08 の動作が壊れない。

この仕組みは汎用的で、「ガード後の反撃」「ダッシュキャンセル」など他のアビリティにも同じパターンで応用できます。
