BP EX 01

コンボ先行入力

UE 5.5 版

BP STEP 08 のコンボに「少し早めに押しても繋がる」先行入力を追加する。判定ロジックはすべて GA の内部に閉じる。

このページのゴール

BP STEP 08 で作ったコンボは、AN_ComboReady が発火してから Q を押す「反応型」でした。 このページでは 先行入力（プリインプット） を追加します。 攻撃モーション中の受付ウィンドウ内に Q を押しておくと、タイミングが来たときに自動で次のコンボへ繋がります。

  • Character BP の Q キーは「イベントを送って Combo1 を起動しようとする」だけ。条件分岐なし。
  • 入力のバッファリングと判定はすべて GA 内部で行う。Character BP 側に変数やタグチェックを置かない。
  • AnimNotify State（範囲通知）で受付ウィンドウを定義し、GA がその範囲内の入力を記憶する。
  • 受付ウィンドウを逃しても、以降のモンタージュ再生中に Q を押せば反応型として繋がるフォールバックも備える。
前提

BP STEP 08 が完成していることが前提です。このページは bp08 の GA・アニメーション・Character BP を改修する形で進めます。

なぜ先行入力が必要か

人間の視覚的反応速度は平均 150〜200ms です。 攻撃アニメーションの当たり判定フレームを「見てから」反応するのは、フレームが短いほどほぼ不可能です。

bp08（反応型） bpEx1（先行入力）
入力のタイミング AN_ComboReady（点通知）が発火した後に Q を押す。受付前の入力は無視される。 ANS_ComboBuffer の Begin〜End（受付ウィンドウ）内に Q を押しておく。End が来た瞬間に自動発動。
難易度 画面を見てから反応が必要。ウィンドウを逃すと再スタート。 アニメーションのリズムに合わせて早めに押せばよい。ANS End 後の反応型フォールバックもある。
GA の構成 Then 1 のみ（ComboReady → Input と直列に待機して TryActivate） Then 1（先行入力バッファ）＋ Then 2（ComboReady → Branch → 先行 or 反応型）の 2 パス

商業タイトルのアクションゲームではほぼすべてで先行入力が実装されています。 このページでは AnimNotify State による入力ウィンドウの定義と、GA 内での入力記憶を組み合わせて実現します。

仕組みを理解する

AnimNotify State とは

bp06 / bp08 で使った AN_AttackHit / AN_ComboReady は特定フレームで一瞬だけ発火する AnimNotify（点）です。 AnimNotify State（範囲）は開始と終了の 2 フレームを持ち、Received Notify Begin と Received Notify End をそれぞれオーバーライドできます。

攻撃開始
モーション開始
← ANS 範囲 →
受付ウィンドウ
この間の Q を GA が記憶
ANS End 以降
反応型フォールバック
この間の Q も GA が処理
モンタージュ終了
GA 終了

ANS_ComboBuffer の Begin（受付ウィンドウ開始）から End（実行タイミング）までに Q を押すと GA がバッファとして記憶します。 End が来た瞬間に記憶があれば即座に次段へ。なければ、その後モンタージュが終わるまで反応型として Q を待ち続けます。

設計の核心：GA が Q の入力イベントを直接受け取る

bp08 では Character BP の cascading TryActivate がコンボの判定を担っていました。 このページでは Send Gameplay Event to Actor を使って Q 入力を Gameplay Event として GA に届けます。 Character BP は「イベントを送る」だけで、どのコンボが発動するかの判断は完全に GA 内で行います。

全体の制御フロー（先行入力パス）
Q Pressed（Character BP） → Send Gameplay Event
Event.Combo.Input → Try Activate Ability By Class
GA_Combo1（コンボ未発動時のみ成功）
GA_Combo1（ANS Begin 後・受付中） → Wait Gameplay Event
Event.Combo.Input Event Received → bComboBuffered = true
GA_Combo1（ANS End = 実行タイミング） → Branch（bComboBuffered）
True → TryActivate GA_Combo2（即座に発動）
False → Wait Gameplay Event (Event.Combo.Input) → TryActivate GA_Combo2（反応型）
False パスでの Wait は GA（モンタージュ）が終了すると自動的にキャンセルされる。

設計する

新しいタグ

タグ 用途 送信者
Event.Combo.BufferStart ANS_ComboBuffer の Begin（受付ウィンドウ開始）が送る Gameplay Event。GA が受け取り、入力の待機を開始する。 ANS_ComboBuffer
Event.Combo.Input Q が押されたときに Character BP が送る Gameplay Event。GA が直接受け取り、バッファセットまたは TryActivate のトリガーとして使う。 Character BP
Event.Combo.Ready は引き続き使用

ANS_ComboBuffer の End は bp08 の AN_ComboReady と同じ Event.Combo.Ready を送信します。GA 側の Then 2 は変更不要です。

GA のクラス変数

bp08 では Character BP に判定変数を置く必要がありましたが、このページでは 各 GA のクラス変数として bComboBuffered（bool）を持ちます。 Character BP をまたいだアクセスが不要になるため、Cast ノードも不要です。

STEP 1 — ANS_ComboBuffer を作る

1
タグを 2 つ新規作成する

任意の GameplayAbility の Class Defaults → AssetTags の「+」から以下の 2 タグを作成します。 作成後、AssetTags の一覧から削除してかまいません。

  • Event.Combo.BufferStart
  • Event.Combo.Input
2
ANS_ComboBuffer Blueprint を作成する

Content Browser で右クリック → Blueprint Class → 親クラスに AnimNotifyState を指定し、 名前を ANS_ComboBuffer とします。

作成した ANS_ComboBuffer を開き、My Blueprint パネル → FUNCTIONS → Override（+）から Received Notify Begin と Received Notify End を両方追加します。

Received Notify Begin

Received Notify Begin → Mesh Component → Get Owner → Send Gameplay Event to Actor
Event Tag: Event.Combo.BufferStart
受付ウィンドウの開始を GA に通知する。

Received Notify End

Received Notify End → Mesh Component → Get Owner → Send Gameplay Event to Actor
Event Tag: Event.Combo.Ready
実行タイミングを GA に通知する。bp08 の AN_ComboReady と同じタグを使うため、GA 側のコードは流用できる。
Actor ピンを必ず繋ぐ

Mesh Component → Get Owner の戻り値を Send Gameplay Event to Actor の Actor ピンに繋いでください。未接続だと ASC が見つからずサイレントに失敗します。

STEP 2 — アニメーションを更新する

3
AM_Combo1 と AM_Combo2 の Notifies を置き換える

AM_Combo1 と AM_Combo2 をそれぞれ開きます。

  1. Notifies トラックの AN_ComboReady を右クリック → Delete（削除）。
  2. Notifies トラックを右クリック → Add Notify State（通知ステートを追加） → ANS_ComboBuffer を選択。
  3. 配置された ANS_ComboBuffer の End 位置（右端）を、削除した AN_ComboReady があったフレームに合わせます。
  4. Begin 位置（左端）を End より 5〜10 フレーム手前に設定します。この区間が受付ウィンドウになります。
Begin 位置の目安

Begin を早めると先行入力できる余裕が増えます。攻撃モーションの前半から受け付けると自然な操作感になります。ANS End 以降のフォールバックもあるため、Begin は比較的早めに設定してかまいません。

AM_Combo3 は変更不要

GA_Combo3 は最後の段なので次のコンボへのバッファは必要ありません。AM_Combo3 は変更しません。

STEP 3 — Character BP を更新する

4
Character BP の Q キーは変更不要

bp08 で Q キーをすでに 2 ノード構成にしている場合、Character BP の変更は不要です。確認のため以下の構成になっているかチェックしてください。

Q Pressed → Send Gameplay Event to Actor
Actor: Self / Event Tag: Event.Combo.Input → Try Activate Ability By Class
GA_Combo1
Send Event → アクティブな GA があれば受け取る（バッファ処理 or 反応型フォールバック）。
TryActivate → どのコンボも動いていなければ Combo1 を起動。動いていれば失敗するだけ。

STEP 4 — GA_Combo1 を改修する

5
bComboBuffered 変数を GA に追加する

GA_Combo1 を開き、My Blueprint パネル → VARIABLES の「+」から変数を追加します。

変数名型デフォルト値
bComboBufferedBooleanfalse

Cast ノードは不要です。GA 自身の変数を直接参照するだけです。

6
GA_Combo1 のグラフを更新する

グラフを以下の構成に更新します。 bp08 からの変更点は 緑の背景 で示します。

Event ActivateAbility → Set bComboBuffered = false → Commit Ability → Sequence
Then 0 → Play Montage and Wait
AM_Combo1 → [各出力] → End Ability
Then 1 → Wait Gameplay Event
Event.Combo.BufferStart / Only Trigger Once: true
Event Received → Wait Gameplay Event
Event.Combo.Input / Only Trigger Once: true Event Received → Set bComboBuffered = true
Then 2 → Wait Gameplay Event
Event.Combo.Ready / Only Trigger Once: true
Event Received → Branch (bComboBuffered)
True → Set bComboBuffered = false → Try Activate Ability By Class（GA_Combo2）
False → Wait Gameplay Event
Event.Combo.Input / Only Trigger Once: true Event Received → Try Activate Ability By Class（GA_Combo2）
Then 1 は「バッファパス」、Then 2 False は「反応型フォールバックパス」。両方とも最終的に TryActivate GA_Combo2 に繋がる。
Then 2 False の Wait は Play Montage and Wait が終了して End Ability が呼ばれると自動的にキャンセルされる。

Then 1 のチェーン（新概念）

Then 1 では Wait Gameplay Event を 2 つ直列に繋いでいます。 最初の Wait（BufferStart）が発火して初めて、次の Wait（Combo.Input）が待機を開始します。 これにより「受付ウィンドウが開く前の Q 入力は完全に無視される」という動作が自然に実現します。

bp08 からの変更点まとめ（GA_Combo1）

追加： ActivateAbility 先頭で bComboBuffered = false（リセット）。 Then 1 新規追加（BufferStart → Input のチェーン Wait → bComboBuffered = true）。 Then 2 の False パスに反応型フォールバック Wait を追加。

削除： Apply Gameplay Effect to Owner（GE_ComboWindow1）（不要になったため）。

GA から TryActivateAbilityByClass を呼ぶ

GA のグラフ内では、右クリック検索で Try Activate Ability By Class を直接配置することはできません（Character BP とは異なり、Target が自動接続されない）。親キャラクターの AbilitySystemComponent 参照を先に取得し、そこからノードを引き出します。

  1. グラフ空白を右クリック → 「Get Avatar Actor from Actor Info」 を検索・配置
  2. 戻り値から右ドラッグ → 「Get Ability System Component」 を検索・配置
  3. 戻り値（AbilitySystemComponent）から右ドラッグ → 「Try Activate Ability By Class」 を検索・配置
  4. In Ability to Activate ピンに GA_Combo2 を指定

STEP 5 — GA_Combo2 と GA_Combo3 を更新する

7
GA_Combo2 を改修する

GA_Combo1 と同じ変更を GA_Combo2 に適用します。異なるのは以下の 3 点のみです。

変更箇所 GA_Combo1 の設定値 GA_Combo2 の設定値
モンタージュ（Then 0） AM_Combo1 AM_Combo2
True パスの TryActivate 先 GA_Combo2 GA_Combo3
False パスのフォールバック TryActivate 先 GA_Combo2 GA_Combo3

GA_Combo2 の Class Defaults → Tags セクションは bp08 の設定のままで変更不要です。

タグ設定の確認

bp08 で以下のように設定済みであれば変更不要です：AssetTags: Ability.Combo.2 / Cancel Abilities with Tag: Ability.Combo.1 / Block Abilities with Tag: Ability.Combo.2 + Ability.Combo.1

Block Abilities with Tag: Ability.Combo.1 を追加する理由

Character BP の Q キーは常に TryActivate GA_Combo1 を試みます。GA_Combo2 が発動中にこれが成功すると GA_Combo1 と GA_Combo2 が同時に走ってしまいます。GA_Combo2 が Ability.Combo.1 をブロックしておくことで、Combo2 中は Combo1 の起動を防ぎます。

8
GA_Combo3 のタグを更新する

GA_Combo3 はグラフ・タグともに変更不要です。最後の段なので次段へのバッファは持ちません。

タグ設定の確認

bp08 で以下のように設定済みであれば変更不要です：AssetTags: Ability.Combo.3 / Cancel Abilities with Tag: Ability.Combo.2 / Block Abilities with Tag: Ability.Combo.3 + Ability.Combo.1 + Ability.Combo.2

動作確認

Play（プレイ）して以下のシナリオを確認します。

  • 先行入力パス：Combo1 再生中の受付ウィンドウ（ANS 範囲）内に Q を押すと、ANS End のタイミングで自動的に Combo2 に切り替わる。
  • 反応型フォールバック：ANS End を過ぎた後（モンタージュ継続中）に Q を押しても Combo2 に繋がる。
  • ウィンドウ外：Combo1 のモンタージュが終了してから Q を押すと Combo1 から再スタートする。
  • 3 連撃：Combo1 → Combo2 → Combo3 と先行入力で全段完走できる。
  • Combo2 中の誤 Combo1 起動なし：Combo2 発動中に Q を押しても Combo1 は起動しない（Block Abilities with Tag が機能している）。
受付ウィンドウの調整

ANS_ComboBuffer の Begin 位置を Sequence エディタで動かすと先行入力できる時間を変えられます。Begin を早めると余裕が増え、遅めると精度が求められます。

まとめ

このページで加えた変更を整理します。

  • Character BP の Q キーは 2 ノードに：Send Gameplay Event と TryActivate GA_Combo1 のみ。条件分岐なし・Cast なし。
  • 判定はすべて GA 内部に：バッファフラグ（bComboBuffered）を GA クラス変数として持ち、受付ウィンドウ内の Q 入力を GA が直接 Gameplay Event として受け取る。
  • チェーン Wait パターン：Wait BufferStart → Wait Combo.Input と直列に繋ぐことで「ウィンドウが開いてから Q を待つ」動作が自然に実現する。
  • 2 パスの設計：先行入力パス（バッファあり→即発動）と反応型フォールバック（バッファなし→その後モンタージュ中に Q を待つ）を両立。GE_ComboWindow によるウィンドウ管理は不要になった。
  • 同時発動の防止：GA_Combo2/3 に Block Abilities with Tag: Ability.Combo.1（および 2）を追加し、Character BP からの TryActivate が誤って下位コンボを起動しないよう保護する。

この「GA がイベントを直接受け取る」パターンは汎用的で、 ガード中の反撃・スキルチェーンなど様々なアビリティ連携に応用できます。
