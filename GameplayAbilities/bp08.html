BP STEP 08

コンボ攻撃（3連撃）

UE 5.5 版

各 GA が内部で次段への繋ぎを待機し、Wait Gameplay Event のチェーンで 3 連撃のフローを制御する。

このステップのゴール

Q を押すたびに 1 段目 → 2 段目 → 3 段目と続くコンボ攻撃を実装します。 各段の GA が「自分の当たり判定フレームでコンボ入力を受けたら次段を起動する」という責任を自分で持つ設計にします。

  • Character BP の Q キーは 2 ノードのみ。コンボのどの段が動くかを Character BP は判断しない。
  • 各 GA が Wait Gameplay Event のチェーンで当たり判定フレームとコンボ入力を内部で待ち、条件が揃ったら自分で次段を TryActivate する。
  • Cancel Abilities with Tag で前段のアビリティを打ち切り、次段に切り替わる。
  • GA_Combo1 / GA_Combo2 / GA_Combo3 の 3 つを作り、3 連撃が機能することを確認する。

仕組みを理解する

Q が押されると Character BP は Gameplay Event を送りつつ GA_Combo1 の起動を試みます。 コンボが進行中であれば GA_Combo1 は起動できませんが、Gameplay Event はアクティブな GA に届きます。 各 GA は内部でその Event を待ち受けており、受け取ったタイミングで次段を自分で起動します。

Q を 3 回タイミングよく押したときの全体の流れ
① Q Pressed → Send Event.Combo.Input + GA_Combo1 発動 → AM_Combo1 再生
↓ AN_ComboReady 発火（Event.Combo.Ready）
GA_Combo1 が Event.Combo.Ready を受け取り、次の Event.Combo.Input を待機する
② Q Pressed → Send Event.Combo.Input → GA_Combo1 が受け取る → GA_Combo2 発動 → GA_Combo1 キャンセル
↓ AN_ComboReady 発火（Event.Combo.Ready）
GA_Combo2 が Event.Combo.Ready を受け取り、次の Event.Combo.Input を待機する
③ Q Pressed → Send Event.Combo.Input → GA_Combo2 が受け取る → GA_Combo3 発動 → GA_Combo2 キャンセル → AM_Combo3 再生して終了
AN_ComboReady が発火する前に Q を押してもタイミング的に繋がらない。AN_ComboReady 後なら次段に繋がる。
AN_ComboReady より前の Q 入力は無視される理由

GA は Wait Gameplay Event (Event.Combo.Ready) が発火して初めて Wait Gameplay Event (Event.Combo.Input) の待機を開始します。ComboReady が来る前に Q を押しても、その時点では GA が Input を待ち受けていないため、何も起きません。

タグを設計する

タグ 用途
Ability.Combo.1 GA_Combo1 の識別タグ。GA_Combo2 の Cancel Abilities with Tag と Block Abilities with Tag から参照される。
Ability.Combo.2 GA_Combo2 の識別タグ。GA_Combo3 の Cancel Abilities with Tag と Block Abilities with Tag から参照される。
Ability.Combo.3 GA_Combo3 の識別タグ。自己ブロックに使う。
Event.Combo.Ready AN_ComboReady が送る Gameplay Event タグ。当たり判定フレームを GA に通知する。
Event.Combo.Input Q が押されたときに Character BP が送る Gameplay Event タグ。アクティブな GA が受け取り、次段へ繋ぐトリガーになる。

STEP 1 — タグと AN_ComboReady を準備する

1
タグを作成して AN_ComboReady を準備する

任意の GameplayAbility の Class Defaults → AssetTags の「+」から以下の 5 つのタグを作成します。 作成後、AssetTags の一覧から削除してかまいません。

  • Ability.Combo.1
  • Ability.Combo.2
  • Ability.Combo.3
  • Event.Combo.Ready
  • Event.Combo.Input

次に Content Browser で AN_AttackHit を右クリック → Duplicate（複製） し、 名前を AN_ComboReady に変更します。

AN_ComboReady を開き、Send Gameplay Event to Actor ノードの Event Tag を Event.Combo.Ready に変更します。 それ以外のノード接続はそのままで構いません。

作成後、AM_Combo1 と AM_Combo2 のコンボ受付を開始したいフレームに AN_ComboReady を Notifies トラックへ配置します（AM_Combo3 への配置は不要です）。

STEP 2 — GA_Combo1 を作る

2
GA_Combo1 の Class Defaults を設定する

Content Browser で右クリック → Blueprint Class → GameplayAbility を親クラスに指定し、 名前を GA_Combo1 とします。 Class Defaults → Tags セクションを以下のように設定します。

フィールド 設定値 意味
AssetTags Ability.Combo.1 このアビリティの識別タグ。
Block Abilities with Tag Ability.Combo.1 発動中は同じ GA_Combo1 を再起動できなくする（自己ブロック）。
3
GA_Combo1 のグラフを組む

BP STEP 06 の Sequence パターンをベースに、 Then 1 に Wait Gameplay Event を 2 つ直列に繋ぐ 形で次段への遷移を作ります。

Event ActivateAbility → Commit Ability → Sequence
Then 0 → Play Montage and Wait
AM_Combo1 → [各出力] → End Ability
Then 1 → Wait Gameplay Event
Event.Combo.Ready / Only Trigger Once: true
Event Received → Wait Gameplay Event
Event.Combo.Input / Only Trigger Once: true Event Received → Try Activate Ability By Class
GA_Combo2
Then 1 の 2 つ目の Wait は Play Montage and Wait が終了して End Ability が呼ばれると自動的にキャンセルされる。
Wait Gameplay Event を直列に繋ぐ

1 つ目の Wait（Event.Combo.Ready）の Event Received ピンから直接 2 つ目の Wait（Event.Combo.Input）を繋ぎます。1 つ目が発火して初めて 2 つ目の待機が始まるため、AN_ComboReady が発火するまでは Q の入力を一切受け付けません。

GA から TryActivateAbilityByClass を呼ぶ

GA のグラフ内では、右クリック検索で Try Activate Ability By Class を直接配置することはできません（Character BP とは異なり、Target が自動接続されない）。親キャラクターの AbilitySystemComponent 参照を先に取得し、そこからノードを引き出します。

  1. グラフ空白を右クリック → 「Get Avatar Actor from Actor Info」 を検索・配置
  2. 戻り値から右ドラッグ → 「Get Ability System Component」 を検索・配置
  3. 戻り値（AbilitySystemComponent）から右ドラッグ → 「Try Activate Ability By Class」 を検索・配置
  4. In Ability to Activate ピンに GA_Combo2 を指定

STEP 3 — GA_Combo2 を作る

4
GA_Combo2 の Class Defaults を設定する

Content Browser → Blueprint Class → GameplayAbility → 名前 GA_Combo2。 Class Defaults → Tags セクションを以下のように設定します。

フィールド 設定値 意味
AssetTags Ability.Combo.2 このアビリティの識別タグ。
Cancel Abilities with Tag Ability.Combo.1 発動時に GA_Combo1 を強制終了して前段のモンタージュを止める。
Block Abilities with Tag Ability.Combo.2（自己）
Ability.Combo.1		 発動中は GA_Combo2 の再起動と GA_Combo1 の起動を防ぐ。
Block Abilities with Tag に Ability.Combo.1 を追加する理由

Character BP の Q キーは常に TryActivate GA_Combo1 を試みます。GA_Combo2 が発動中にこれが成功すると GA_Combo1 と GA_Combo2 が同時に走ってしまいます。Ability.Combo.1 をブロック対象に加えることで、Combo2 中は Combo1 の起動を防ぎます。

5
GA_Combo2 のグラフを組む

グラフは GA_Combo1 と同じ構成です。モンタージュ名と TryActivate 先のみ変えます。

Event ActivateAbility → Commit Ability → Sequence
Then 0 → Play Montage and Wait
AM_Combo2 → [各出力] → End Ability
Then 1 → Wait Gameplay Event
Event.Combo.Ready / Only Trigger Once: true
Event Received → Wait Gameplay Event
Event.Combo.Input / Only Trigger Once: true Event Received → Try Activate Ability By Class
GA_Combo3

STEP 4 — GA_Combo3 を作る

6
GA_Combo3 を作成して設定する

Content Browser → Blueprint Class → GameplayAbility → 名前 GA_Combo3。 Class Defaults を以下のように設定します。

フィールド 設定値
AssetTags Ability.Combo.3
Cancel Abilities with Tag Ability.Combo.2
Block Abilities with Tag Ability.Combo.3（自己）/ Ability.Combo.1 / Ability.Combo.2

グラフは Then 1 なしのシンプルな構成です。3 段目は最後の攻撃なので、次段への遷移は不要です。

Event ActivateAbility → Commit Ability → Play Montage and Wait
AM_Combo3 → [各出力] → End Ability

STEP 5 — キャラクター BP を更新する

7
BeginPlay に Give Ability を追加する

キャラクター Blueprint の Event BeginPlay グラフを開きます。 既存の Give Ability の続きに GA_Combo1・GA_Combo2・GA_Combo3 の 3 つを追加します。

Event BeginPlay → ... Give Ability（GA_Combo1） → Give Ability（GA_Combo2） → Give Ability（GA_Combo3）
3 つとも Give しないと TryActivateAbilityByClass が動作しない。GA_Combo2 / GA_Combo3 は GA 内から呼ぶが、Give は必要。
既存の GA_Attack について

BP STEP 05〜07 で使っていた GA_Attack の Q キーのノードはこのステップでは使用しません。コンボ用の Q キーロジックに置き換えます。ただし GA_Attack の Give Ability ノードは削除せずそのまま残してください。BP EX 02 でヒット検出の実装先として使います。

8
Q キーを 2 ノードに更新する

Q キーのイベントから 2 つのノードのみ 繋ぎます。コンボのどの段を起動するかは Character BP が判断しません。

Q Pressed → Send Gameplay Event to Actor
Actor: Self / Event Tag: Event.Combo.Input → Try Activate Ability By Class
GA_Combo1
Send Event → アクティブな GA_Combo1 または GA_Combo2 があれば受け取り、次段を起動する。
TryActivate → どのコンボも動いていなければ Combo1 を起動。動いていれば失敗するだけ（何も起きない）。

動作確認

Play（プレイ）して以下を確認します。

  • Q を押すと 1 段目（AM_Combo1）が再生される
  • AN_ComboReady 配置フレーム以降に Q を押すと 2 段目（AM_Combo2）に切り替わり、1 段目が即座に停止する
  • 同様に 2 段目の AN_ComboReady 以降に Q を押すと 3 段目（AM_Combo3）に切り替わる
  • AN_ComboReady より早いタイミングで Q を押しても次段に進まない（GA が ComboReady を待っているため）
  • Q を連打しても 3 段目以降は発動しない（GA_Combo3 が Block Abilities with Tag で自己ブロック）
なぜモンタージュを 3 つに分けるのか

ActionRPG などのサンプルでは 1 モンタージュをセクションで分割し、1 つの GA がセクションをジャンプしながらコンボを進めるアプローチもあります。アニメーション担当が 1 アセットで全段を管理しやすい利点があります。

このステップでは各段を独立した GA にすることで「パンチ→パンチ→蹴り」「蹴り→パンチ→パンチ」のような別コンボパターンも、アビリティの組み合わせを差し替えるだけで作れる拡張性を持たせています。

まとめ

  • GA が次段の起動を自分で担う：Character BP はコンボの段数や条件を知らなくてよい。各 GA が自分の後に来る GA を TryActivate する責任を持つ。
  • Wait Gameplay Event の直列チェーン：ComboReady → Combo.Input と順番に待つことで「当たり判定フレームに到達してから Q を受け付ける」という制御が自然に実現する。
  • Cancel Abilities with Tag：次段が発動した瞬間に前段を強制終了する。モンタージュの切り替えはこれが担う。
  • Block Abilities with Tag のクロスブロック：GA_Combo2/3 が下位段のタグもブロック対象に加えることで、Character BP からの TryActivate Combo1 が誤って割り込まないよう保護する。

次の BP EX 01 では、この設計に 先行入力（当たり判定フレームより前に Q を押しておいても繋がる仕組み）を追加します。 AnimNotify State と GA 内の入力バッファ変数を組み合わせることで実現します。
