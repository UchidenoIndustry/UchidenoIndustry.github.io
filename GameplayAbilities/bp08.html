BP STEP 08

コンボ攻撃（3連撃）

UE 5.5 版

Duration GE でコンボウィンドウを管理し、Activation Required Tags で 3 連撃のフローを制御する。

このステップのゴール

Q を押すたびに 1 段目 → 2 段目 → 3 段目と続くコンボ攻撃を実装します。 ただし「前の攻撃の当たり判定フレームで Q を押した場合のみ次段に進める」という タイミング制限を加えます。

  • 新概念 Activation Required Tags でコンボウィンドウを管理する
  • Cancel Abilities with Tag で前段のアビリティを打ち切り、次段に切り替える
  • Q キーで高い番号から順に TryActivateAbilityByClass を試す cascading パターンを実装する
  • GA_Combo1 / GA_Combo2 / GA_Combo3 を作り、3 連撃が機能することを確認する

このステップで登場する Activation Required Tags は、 BP STEP 04 で学んだ Activation Blocked Tags の逆版です。 指定タグが ASC にある間だけアビリティを発動できます。 コンボウィンドウが開いている間だけ次段が発動できる、という制御に使います。

仕組みを理解する

コンボの核心は コンボウィンドウ という短い時間枠です。 モンタージュの任意のフレームに配置した AN_ComboReady が発火すると Duration GE が付与され、 「この GE が ASC に存在する間（約 0.5 秒）だけ次の段が発動できる」という仕組みで タイミングを要求します。

Q を 3 回タイミングよく押したときの全体の流れ
① Q Pressed → GA_Combo1 発動 → AM_Combo1 再生
↓ AN_ComboReady 発火
GE_ComboWindow1 付与
Status.ComboWindow.1 を 0.5 秒間 ASC に付ける
② Q Pressed → Combo3 → 失敗
.2 がないため → GA_Combo2 発動
.1 があるため成功 → GA_Combo1 キャンセル
↓ AN_ComboReady 発火
GE_ComboWindow2 付与
Status.ComboWindow.2 を 0.5 秒間 ASC に付ける
③ Q Pressed → GA_Combo3 発動 → GA_Combo2 キャンセル → AM_Combo3 再生して終了
ウィンドウ外で Q を押すと Combo2/3 は発動できないため、Combo1 から再スタートする。

タグを設計する

タグ 用途
Ability.Combo.1 GA_Combo1 の識別タグ。GA_Combo2 の Cancel Abilities with Tag から参照される。
Ability.Combo.2 GA_Combo2 の識別タグ。GA_Combo3 の Cancel Abilities with Tag から参照される。
Ability.Combo.3 GA_Combo3 の識別タグ。自己ブロックに使う。
Status.ComboWindow.1 Combo1 の当たりフレームで開くウィンドウ。GE_ComboWindow1 が付与する。GA_Combo2 の Activation Required Tags に設定する。
Status.ComboWindow.2 Combo2 の当たりフレームで開くウィンドウ。GE_ComboWindow2 が付与する。GA_Combo3 の Activation Required Tags に設定する。
タグの階層について

Ability.Combo.1 のように3階層にすると、Ability.Combo という親タグで「コンボ系アビリティ全体」をまとめて参照することもできます。今回は個別のタグのみ使います。

GE_ComboWindow を作る

1
タグと GE を 2 つ作成する

まず任意の GameplayAbility の Class Defaults → AssetTags の「+」からタグを5つ新規作成します。

  • Ability.Combo.1
  • Ability.Combo.2
  • Ability.Combo.3
  • Status.ComboWindow.1
  • Status.ComboWindow.2
  • Event.Combo.Ready

次に GE を 2 つ作成します。 Content Browser で右クリック → Blueprint Class → 親クラス GameplayEffect で GE_ComboWindow1 と GE_ComboWindow2 をそれぞれ作成します。

各 GE の Class Defaults を以下のように設定します。

設定箇所 GE_ComboWindow1 GE_ComboWindow2
Duration Policy Has Duration Has Duration
Duration Magnitude 0.5 0.5
Components → Grant Tags → Add to Inherited Status.ComboWindow.1 Status.ComboWindow.2
なぜ GE を分けるのか

GE を 1 つ（Status.ComboWindow）にすると、Combo1 のウィンドウが開いた時点で GA_Combo2 だけでなく GA_Combo3 の条件も満たされてしまい、Combo1 → Combo3 と段を飛ばして発動してしまいます。 GE を段ごとに分けて別々のタグを使うことで「Combo1 のウィンドウでは Combo2 だけ、Combo2 のウィンドウでは Combo3 だけ」が発動できる状態を正確に作れます。

AN_ComboReady を作る

2
AN_ComboReady を作成する

Content Browser で AN_AttackHit を右クリック → Duplicate（複製） し、 名前を AN_ComboReady に変更します。

AN_ComboReady を開き、Send Gameplay Event to Actor ノードの Event Tag を Event.Combo.Ready に変更します。 それ以外のノード接続はそのままで構いません。

作成後、AM_Combo1 と AM_Combo2 のコンボ受付を開始したいフレームに AN_ComboReady を Notifies トラックへ配置します。

GA_Combo1 を作る

3
GA_Combo1 Blueprint を作成して Class Defaults を設定する

Content Browser で右クリック → Blueprint Class → GameplayAbility を親クラスに指定し、 名前を GA_Combo1 とします。

Class Defaults → Tags セクションを以下のように設定します。

フィールド 設定値
AssetTags (Default AbilityTags) Ability.Combo.1
Block Abilities with Tag Ability.Combo.1（自己再発動防止）
GA_Combo1 に Activation Required Tags は不要

1 段目はコンボウィンドウなしで発動できる必要があるため、Activation Required Tags は設定しません。

4
GA_Combo1 のグラフを組む

BP STEP 06 と同じ Sequence パターン を使います。 モンタージュを再生しながら、当たり判定フレームで GE_ComboWindow を付与します。

Event ActivateAbility → Commit Ability → Sequence
Then 0 → Play Montage and Wait
Montage: AM_Combo1
On Completed → End Ability
On Blend Out → End Ability
On Interrupted → End Ability
On Cancelled → End Ability
Then 1 → Wait Gameplay Event
Event.Combo.Ready / Only Trigger Once: true
Event Received → Apply Gameplay Effect
to Owner
GE_ComboWindow1
Wait Gameplay Event が発火すると GE_ComboWindow1 が付与され、0.5 秒間 Status.ComboWindow.1 が ASC に付く。
Apply Gameplay Effect to Owner の場所

GA Blueprint のイベントグラフで右クリック → 「Apply Gameplay Effect to Owner」 を検索します。見つからない場合は 「Apply Gameplay Effect to Self」 も試してみてください。Gameplay Effect Class に GE_ComboWindow1、Level に 1.0 を設定します。

GA_Combo2 を作る

5
GA_Combo2 Blueprint を作成して Class Defaults を設定する

Content Browser → Blueprint Class → GameplayAbility → 名前 GA_Combo2。 Class Defaults → Tags セクションを以下のように設定します。

フィールド 設定値 意味
AssetTags Ability.Combo.2 このアビリティのラベル。
Activation Required Tags Status.ComboWindow.1 新概念：ASC に Status.ComboWindow.1 がある間だけ発動できる。Combo1 のウィンドウ（GE_ComboWindow1）が切れると発動できなくなる。
Cancel Abilities with Tag Ability.Combo.1 発動時に Ability.Combo.1 タグを持つアビリティ（GA_Combo1）を強制キャンセルする。
Block Abilities with Tag Ability.Combo.2 自己再発動防止。
Cancel Abilities with Tag のポイント

GA_Combo2 が発動した瞬間に GA_Combo1 は On Cancelled ピンから End Ability が呼ばれます。モンタージュも即座に停止し、GA_Combo1 の Activation Owned Tags などもすべてクリアされます。

6
GA_Combo2 のグラフを組む

グラフは GA_Combo1 と同じ Sequence パターンです。モンタージュ名と Wait Gameplay Event の後の処理が同じです。

Event ActivateAbility → Commit Ability → Sequence
Then 0 → Play Montage and Wait
Montage: AM_Combo2 → [各出力] → End Ability
Then 1 → Wait Gameplay Event
Event.Combo.Ready / Only Trigger Once: true Event Received → Apply Gameplay Effect
to Owner (GE_ComboWindow2)
GA_Combo2 の当たり判定フレームで GE_ComboWindow2 を付与し、Status.ComboWindow.2 を ASC に付けて 3 段目へのウィンドウを開く。

GA_Combo3 を作る

7
GA_Combo3 Blueprint を作成して設定する

Content Browser → Blueprint Class → GameplayAbility → 名前 GA_Combo3。 Class Defaults を以下のように設定します。

フィールド 設定値
AssetTags Ability.Combo.3
Activation Required Tags Status.ComboWindow.2
Cancel Abilities with Tag Ability.Combo.2
Block Abilities with Tag Ability.Combo.3

グラフは シンプルな Play Montage and Wait のみ です。 3 段目は最後の攻撃なので、コンボウィンドウを開く必要はありません。 Wait Gameplay Event と GE_ComboWindow の適用は不要です。

Event ActivateAbility → Commit Ability → Play Montage and Wait
Montage: AM_Combo3
On Completed → End Ability
On Blend Out → End Ability
On Interrupted → End Ability
On Cancelled → End Ability

キャラクター BP を更新する

8
BeginPlay に Give Ability を追加する

キャラクター Blueprint の Event BeginPlay グラフを開きます。 既存の Give Ability ノードの続きに GA_Combo1・GA_Combo2・GA_Combo3 の 3 つを追加します。

Event BeginPlay → Give Ability
GA_Combo1 → Give Ability
GA_Combo2 → Give Ability
GA_Combo3
3 つとも Give しないと TryActivateAbilityByClass が動作しない。
既存の GA_Attack について

BP STEP 05〜07 で使っていた GA_Attack の Give Ability と Q キーのノードは、このステップでは使用しません。コンボ用の Q キーロジックに置き換えます。既存のノードはそのままにするか、削除してください。

9
Q キーのロジックを cascading TryActivate に更新する

Q キーのイベントノードから、高い番号から順に TryActivateAbilityByClass を呼びます。 戻り値（Return Value）を Branch（分岐）に繋ぎ、 false（発動失敗）のときだけ次の番号を試します。

Q Pressed → Try Activate Ability By Class
GA_Combo3 → Return Value → Branch
True（発動成功）→ 何もしない
False → Try Activate Ability By Class
GA_Combo2 → Return Value → Branch
True → 何もしない
False → Try Activate Ability By Class
GA_Combo1
コンボウィンドウが開いていれば高い番号が優先される。ウィンドウがなければ Combo3/2 は Activation Required Tags で弾かれ、Combo1 から始まる。
なぜ高い番号から試すのか

Combo1 のウィンドウ中は Status.ComboWindow.1 だけが存在します。Combo3（Status.ComboWindow.2 要求）は失敗し、Combo2（Status.ComboWindow.1 要求）が成功します。Combo2 のウィンドウ中は Status.ComboWindow.2 が存在するため Combo3 が成功します。低い番号から試すと Combo1 が常に発動してしまい順番に進みません。

動作確認

Play（プレイ）して以下を確認します。

  • Q を押すと 1 段目（AM_Combo1）が再生される
  • 当たり判定フレームのタイミングで Q を押すと 2 段目（AM_Combo2）に切り替わり、1 段目が即座に停止する
  • 同様に 2 段目の当たり判定フレームで Q を押すと 3 段目（AM_Combo3）に切り替わる
  • ウィンドウを外れた（遅い）タイミングで Q を押すと 1 段目からやり直しになる
  • Q を連打しても 3 段目以降は発動しない（GA_Combo3 が Block Abilities with Tag で自己ブロック）
GE_ComboWindow の持続時間の調整

0.5 秒では厳しすぎる・または緩すぎる場合は GE_ComboWindow1 / GE_ComboWindow2 の Duration Magnitude を変更してください。ウィンドウの長さはアニメーションのテンポに合わせて調整します。

なぜモンタージュを 3 つに分けるのか

ActionRPG などのサンプルでは、1 つのモンタージュをセクション（Section 1 / Section 2 / Section 3）で分割し、1 つの GA がセクションをジャンプしながらコンボを進めるアプローチもあります。アニメーション担当が 1 アセットで全段を管理しやすい利点があります。

このステップでは GAS のタグシステム（Activation Required Tags / Cancel Abilities with Tag）を実感することが目的のため、各段を独立したアビリティとして分割しています。アビリティを分けることで「条件が揃ったときだけ次の段を発動する」という GAS らしい制御の流れが見えやすくなります。また、各段が独立した部品になっているため、「パンチ→パンチ→蹴り」「蹴り→パンチ→パンチ」のような別コンボパターンも、アビリティの組み合わせを差し替えるだけで作れます。

まとめ

このステップで Blueprint 編の全ての要素が揃いました。

  • Activation Required Tags：ASC に指定タグがある間だけ発動できる。コンボウィンドウ制御のように「条件が揃ったときだけ」という発動制限に使う。
  • Cancel Abilities with Tag：発動時に、指定タグを持つ他のアビリティを強制終了する。コンボの段切り替えに使う。
  • cascading TryActivate：高い番号から順に TryActivate を試すパターン。段ごとに異なるウィンドウタグ（Status.ComboWindow.1 / .2）を使うことで、各段が自分の直前ウィンドウでのみ発動できるよう制御する。

Blueprint 編ではここまで、ASC・GameplayAbility・GameplayTag・GameplayEffect・Ability Task の5要素をすべて使ってきました。 ダメージの数値管理（AttributeSet）や、より複雑なゲームプレイ効果の実装には C++ が必要です。
